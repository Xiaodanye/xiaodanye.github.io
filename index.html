Vue虚拟DOM-diff算法

虚拟DOM & diff算法

在前端性能优化的一个秘诀就是尽可能的减少DOM的操作，不仅仅是DOM相对较慢，更是因为变动DOM会造成浏览器的回流和重绘，这些都会降低性能，因此，我们需要虚拟DOM，在patch（比较新旧虚拟DOM更新去更新视图）过程中尽可能的一次性将差异更新到DOM中，这样就保证了DOM不会出现了性能很差的情况。

使用 虚拟DOM 改变了当前的状态不需要立即去更新DOM，而是对更新的内容进行更新，对于没有改变的内容不做任何处理，通过前后两次差异进行比较。

虚拟DOM

Virtual DOM 其实就是一棵以 JavaScript 对象（VNode 节点）作为基础的树，用对象属性来描述节点，相当于在js和真实dom中间加来一个缓存，利用diff算法避免没有必要的dom操作，从而提高性能。

底层的实现可以简单理解为三个步骤

  1. 用JavaScript对象结构表述dom树的结构，然后用这个树构建一个真正的dom树，插到浏览器的页面中。
  2. 当状态改变了，vue 便会重新构造一棵树的对象树，然后用这个新构建出来的树和旧树进行对比（只进行同层对比），记录两棵树之间的差异。
  3. 把记录的差异在重新应用到真正的dom树，视图就更新了。

模版（template）编译过程

  • parse：会使用正则等方式解析template模版中的指令、class、style等数据，形成抽象语法树(AST)

  • optimize：对抽象语法树(AST) 进行静态节点标记，减少不必要的比较和计算，优化后文diff算法

  • generate：抽象语法树(AST) 生成 render 函数代码字符串，等待后面的调用

diff算法（snabbdom）

当 Virtual DOM 中状态改变了，vue便会重新构造一棵树的对象树，然后用这个新构建出来的树和旧树进行对比。

核心概念

  1. 只比较同一级，不跨级比较
  2. 只有是同一个虚拟节点(tag & key)，diff算法才进行精细化比较，否则直接删掉重建，不再深度比较

具体实现（patch & updateChildren）

在 patch方法 中，有两种使用场景：1-第一个参数为元素。作用是将 Virtual DOM 挂载到该元素节点上。2-第一个参数是 Virtual DOM 。此时作用是比较新旧 Virtual DOM，将二者差异更新到视图上。首先比较新旧虚拟节点是否是同一个节点，如果不是同一个节点，那么就会将旧的节点删除掉，插入新的虚拟节点，然后再使用 createElement方法 创建真实DOM，渲染到真实的 DOM树。如果是同一个节点，使用 patchVnode方法 比较新旧节点，包括属性更新、文本更新、子节点更新。节点内容有两种类型：文本节点 & 子节点。新旧节点均有子节点，则需要进行 diff算法，调用updateChildren方法；否则直接用新节点的内容替换旧节点的内容。

在updateChildren方法中，首先将新旧节点的子节点都提取出来，然后使用的是 双指针 的方式进行四种优化策略循环比较。分别是：新前与旧前比较、新后与旧后比较、新后与旧前比较、新前与旧后比较。如果四种优化策略方法均没有命中，则会进行遍历方法进行比较（源码中使用了Map对象进行了缓存，加快了比较的速率），如果设置了 key，就会使用 key 进行比较，找到当前的新节点的子节点在 Map 中的映射位置，如果不存在，则需要添加节点，存在则需要移动节点。最后，循环结束之后，如果新节点还有剩余节点，则说明需要添加节点，如果旧节点还有剩余节点，则说明需要删除节点。

PS: v-for 中 key 的作用?

  1. key 是当前节点的唯一标识
  2. Vue 在 patch 过程中判断两个节点是否是相同节点的依据：元素类型 和 key。

参照：Vue 虚拟DOM 和 Diff 算法

Vue父子组件

父子组件传值

方式一：props & event

在Vue中，父子组件的关系可以总结为props向下传递，事件向上传递。父组件向子组件传递props数据，子组件通过触发事件向父组件回传数据。

兄弟组件之间的传值：兄弟组件之间的传值和父子组件之间的传值非常相似，都是通过$emit。通过新建一个新的vue实例，连接着两个组件，也就是一个中央事件总线Event Bus。

  1. 首先创建一个事件总线，例如Event Bus，作为一个通讯的桥梁；
  2. 在需要传值的组件中，通过 $emit 触发一个自定义事件，并传递参数；
  3. 在接收数据的组件中，通过 $on 监听自定义事件，并处理传递过来的参数；

方式二：ref

ref 属性可定义在子组件或原生DOM上，如果在子组件上，则指向子组件实例，如果在原生DOM上，则指向原生DOM元素（可以用做元素选择，省去querySelector的烦恼）。

传递数据的思路：在父组件内通过 ref 获取子组件实例，然后调用子组件方法，并传递相关数据作为参数。

方式三：children & parent

与 ref 类似，例如 $children 可以返回一个子组件集合，通过下标可以访问各个子组件的实例，然后进行操作。

方式四：provide & inject（官方不推荐在生产环境使用）

provide意为提供，当一个组件通过provide提供了一个数据，那么它的子孙组件就可以使用inject接受注入，从而可以使用祖先组件传递过来的数据。

方式五：vuex

vue官方推荐的全局状态管理插件，可以理解为一个全局的空间，所有的组件都可以通过特定的方式获取到数据。Vuex 是什么？

参照：Vue中父组件向子组件传递数据的几种方法; VUE父子组件之间的传值，以及兄弟组件之间的传值

Vue双向绑定

双向绑定

双向绑定依赖于数据劫持 + 发布-订阅者的方式，包括两个方面：数据变化更新视图，视图变化更新数据。视图变化更新数据可以通过事件监听实现，比如 input 标签监听 input 事件就可以实现了。重点在于数据变化更新视图。

当一个 Vue 实例创建时，Vue 会遍历 data 选项的属性，通过 Object.defineProperty() 重写 getter/setter 实现数据劫持；并且在内部追踪相关依赖（订阅者），在属性被修改时通知变化（监听者）。具体实现通过为每个属性分配一个订阅者集合的管理数组，然后在编译的时候在访问相关属性会触发 get 方法，在 get 方法中向该属性的数组中添加订阅者（理论上只要用到该属性的指令：v-model、v-bind以及字符串模版）。之后当相关属性发生变更时，会触发该属性的 set 方法，在 set 方法内发布通知订阅者数组，订阅者数组循环调用各订阅者的 update 方法更新视图。

参照：Vue数据双向绑定；Vue数据双向绑定

PS：实现双向绑定 Proxy 与 Object.defineProperty 相比优劣如何?

  1. Object.definedProperty 的作用是劫持一个对象的属性，劫持属性的 getter 和 setter 方法，在对象的属性发生变化时进行特定的操作。而 Proxy 劫持的是整个对象。
  2. Proxy 会返回一个代理对象，我们只需要操作新对象即可，而 Object.defineProperty 只能遍历对象属性直接修改。
  3. Object.definedProperty 不支持数组，更准确的说是不支持数组的各种API，因为如果仅仅考虑arry[i] = value 这种情况，是可以劫持的，但是这种劫持意义不大。而 Proxy 可以支持数组的各种API。
  4. 尽管 Object.defineProperty 有诸多缺陷，但是其兼容性要好于 Proxy 。

Vue-SPA

SPA

SPA（single-page application）仅在 Web 页面初始化时加载相应的 HTML、JavaScript 和 CSS。一旦页面加载完成，SPA 不会因为用户的操作而进行页面的重新加载或跳转；取而代之的是利用路由机制实现 HTML 内容的变换，数据则需要通过ajax请求获取。

单页应用与多页应用的区别

单页面应用（SPA） 多页面应用（MPA）
组成 一个主页面和多个页面片段 多个主页面
刷新方式 局部刷新 整页刷新
url模式 哈希模式 历史模式
SEO搜索引擎优化 难实现，可使用SSR方式改善 容易实现
数据传递 容易 通过url、cookie、localStorage等传递
页面切换 速度快，用户体验良好 切换加载资源，速度慢，用户体验差
维护成本 相对容易 相对复杂

优劣：

优点：用户体验好；对服务器压力小；前后端分离（前端逻辑，后端数据）；

劣势：

  • 初次加载耗时多：为实现单页 Web 应用功能及显示效果，需要在加载页面的时候将 JavaScript、CSS 统一加载，部分页面按需加载；
  • 不利于 SEO：由于所有的内容都在一个页面中动态替换显示，所以在 SEO 上其有着天然的弱势。

实现：核心通过监听url中的hash来进行路由跳转

  1. 监听地址栏中hash变化驱动界面变化
  2. 用pushsate记录浏览器的历史，驱动界面发送变化

PS: 如何给SPA做SEO：SSR服务端渲染；静态化；使用Phantomjs针对爬虫处理

参照：什么是SPA单页面应用；

Vue-computed_methods_watch

computed & methods & watch

methods：函数方法

  • methods 里面定义的是函数，适合用于业务逻辑处理
  • methods 每次需要重新加载，调用需要执行函数

computed：计算属性（一个数据受多个数据影响）

  • computed 是计算属性，与 data 对象里的数据属性类似，用来防止文本插值中逻辑过重，导致不易维护
  • computed 支持缓存数据，如果函数所依赖的属性没有发生变化，那么调用当前的函数的时候会从缓存中读取
  • computed 可以完成各种复杂的逻辑，但无法传递参数，函数最后必须有 return 返回值

watch：属性监听（一个数据影响多个数据）

  • 当想要执行异步或者路由变化时使用watch，函数名称必须要和data中的属性名一致

  • watch只会监听数据的值是否发生改变，而不会去监听数据的地址是否发生改变（引用无效）

  • watch 不支持数据缓存，在每次监听的值发生变化的时候都会执行回调

  • watch 中的函数有两个参数，前者是newVal，后者是oldVal

  • 页面首次加载的时候做一次监听 immediate: true

总结：

  1. methods，watch 和 computed 都是以函数为基础的，但各自却都不同
  2. methods 里面是用来定义函数的，它需要手动调用才能执行。而不像 watch 和 computed 那样，“自动执行”预先定义的函数，相比于 watch / computed，methods 不处理数据逻辑关系，只提供可调用的函数
  3. watch 和 computed 都是以 Vue 的依赖追踪机制为基础的，当某一个数据发生变化的时候，所有依赖这个数据的“相关”数据“自动”发生变化，也就是自动调用相关的函数去实现数据的变动

参照：Vue核心知识：computed、methods和watch的区别; vue中Computed、Methods、Watch区别

Vue-v-if_v-for

v-for & v-if

从源码的角度（compiler/codegen/index.js）

  • 在 vue 2.x 中，在一个元素上同时使用 v-if 和 v-for 时，v-for 会优先作用。
  • 在 vue 3.x 中，v-if 总是优先于 v-for 生效。

v-for 和 v-if 不应该一起使用，如果两者同时出现的话，那每次循环都会执行 v-if，会很浪费性能。正确的做法应该是再 v-for 的外面新增一个模板标签template，在template上使用 v-if。

我们经常会遇见这种情况，在 v-for 的时候，在数组中会有一些需要进行 v-if 的判断，这种情况下，我们如果先使用computed将不需要渲染的项过滤出来，那么在进行 v-for 的时候，循环的就只是需要渲染的项，这也是提升vue性能的一种方式。

参照：v-if和v-for的优先级

Vue-v-if_v-show

v-if & v-show

vue中显隐方法常用两种，v-if 和 v-show，但这两种是有区别的。

  1. 实现手段：v-if是动态的向DOM树内添加或者删除DOM元素来控制元素的显隐；v-show 是通过设置DOM元素的display样式：block为显示，none为隐藏；
  2. 编译过程：v-if 切换有一个局部编译/卸载的过程，切换过程中合适地销毁和重建内部的事件监听和子组件；v-show只是简单的基于css切换；
  3. 编译条件：v-if 是惰性的，如果初始条件为假，则什么也不做，只有在条件第一次变为真时才开始局部编译； v-show 是在任何条件下（首次条件是否为真）都被编译，然后被缓存，而且DOM元素保留；
  4. 性能消耗：一般来说，v-if 有更高的切换消耗而 v-show 有更高的初始渲染消耗。因此，如果需要频繁切换 v-show 较好，如果在运行时条件不大可能改变 v-if 较好。

Vue-v-model

v-model

v-model是Vue中双向绑定的一个指令，可以绑定一个动态值到视图，同时视图中变化能改变该值。v-model是语法糖，监听用户的输入事件，并及时更新数据。默认情况下相于:value和@input。

通常在表单项上使用v-model： <input>、<textarea> 及 <select> 元素上创建双向数据绑定

v-model 在内部为不同的输入元素使用不同的属性并抛出不同的事件：@change 或者 @input

PS：v-model 会忽略所有表单元素的 value、checked、selected 的初始值，而是总是将当前活动实例的数据作为数据来源

Vue-scoped_样式穿透

scoped & 式样穿透

定义：在vue文件中的style标签上，有一个特殊的属性：scoped。scoped设计的初衷就是不能让当前组件的样式修改其他任何地方的样式。当一个style标签拥有scoped属性时，它的CSS样式就只能作用于当前的组件，也就是说，该样式只能适用于当前组件元素。通过该属性，可以使得组件之间的样式不互相污染。如果一个项目中的所有style标签全部加上了scoped，相当于实现了样式的模块化。

Scope CSS 的本质是基于HTML 和 CSS 属性选择器，即分别给 HTML 标签和 CSS 选择器添加 data-v-xxx(唯一)

具体来说，它是通过 vue-loader 实现的，实现过程大致分 3 步：

  1. 首先 vue-loader 会解析 .vue 组件，提取出 template、script、style 对应的代码块；
  2. 然后构造组件实例，在组件实例的选项上绑定 Scoped-Id；
  3. 最后对 style 的 CSS 代码进行编译转化，应用 Scoped-Id 生成选择器的属性；

PS：如果组件内部包含有其他组件，只会给其他组件的最外层标签加上当前组件的Scoped-Id

样式穿透：默认父组件添加scoped则父组件无法修改子组件的样式（即使通过权重也不行），如果需要在局部组件中修改引入的第三方组件库的样式，又不想去除scoped属性造成组件之间的样式覆盖，可以通过特殊的方式穿透scoped：

  1. 采用全局属性和局部属性混合的方式
  2. 每个组件在最外层添加一个唯一的class区分不同的组件
  3. 使用深层选择器deep

Vue-nextTick

nextTick

首先我们要对 Vue 的数据更新有一定理解： Vue 是依靠数据驱动视图更新的，该更新的过程是异步的，即修改数据时，视图不会立即更新，而是会开启1个队列，监听数据变化，并缓存在同一事件循环中发生的所有数据变化（如果同一个 watcher 被多次触发，只会被推入到队列中一次，这种在缓冲时去除重复数据对于避免不必要的计算和 DOM 操作是非常重要的），最后统一进行视图更新。为了确保得到更新后的DOM，所以设置了 Vue.nextTick()方法。

Vue.nextTick([callback, context])用于在下次 DOM 更新循环结束之后执行延迟回调，它接受2个参数（回调函数和执行回调函数的上下文环境）。回调函数会在 DOM 更新完成后就会调用，回调函数中的执行的应该是会对 DOM 进行操作的 js 代码。

  • {Function} [callback]：回调函数，不传时提供promise调用
  • {Object} [context]：回调函数执行的上下文环境，不传默认是自动绑定到调用它的实例上。

应用场景：

  • 在 Vue 生命周期的created()钩子函数进行的 DOM 操作一定要放在Vue.nextTick()的回调函数中。

    原因：created()钩子函数执行时 DOM 其实并未进行渲染。

  • 在数据变化后要执行的某个操作，而这个操作需要使用随数据改变而改变的 DOM 结构的时候，这个操作应该放在Vue.nextTick()的回调函数中。

    原因：数据与视图的数据尚未完成同步，此时操作DOM元素的得到的是更新前的对象。

参照：Vue.nextTick 的原理和用途; 简单理解Vue中的nextTick
